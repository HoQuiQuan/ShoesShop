API nội bộ (Internal API) là các API được xây dựng chỉ để sử dụng bên trong hệ thống hoặc tổ chức, không public ra bên ngoài cho người dùng hoặc bên thứ ba.

🔹 Hiểu đơn giản

  • API nội bộ = “cổng giao tiếp riêng” giữa các phần trong hệ thống

  • Chỉ các service, backend, hoặc app nội bộ mới gọi được

🔹 Ví dụ thực tế

Giả sử bạn làm web bán hàng:

🧩 Hệ thống có nhiều phần:

  • Frontend (Next.js)

  • Backend (NestJS)

  • Service thanh toán

  • Service quản lý user

👉 API nội bộ sẽ là:

  • /internal/users/get-by-id

  • /internal/orders/create

  • /internal/payment/process

➡️ Những API này:

  • ❌ Không cho client (trình duyệt) gọi trực tiếp

  • ✅ Chỉ backend hoặc service khác gọi

🔹 So sánh với API public

Loại APIAi dùngVí dụ
API nội bộDev trong hệ thống/internal/user/profile
API publicNgười dùng / bên ngoài/api/products

🔹 Đặc điểm của API nội bộ

  • 🔒 Bảo mật cao (không expose ra internet)

  • ⚡ Nhanh hơn (không cần validate quá nhiều như public API)

  • 🧱 Dùng trong kiến trúc microservices

  • 🎯 Phục vụ logic nội bộ (business logic)

🔹 Ví dụ code (NestJS)

TypeScript
@Controller('internal/users')
export class InternalUserController {
  @Get(':id')
  getUserById(@Param('id') id: string) {
    return this.userService.findById(id);
  }
}

➡️ API này thường:

  • được gọi từ service khác

  • hoặc qua gateway

🔹 Khi nào dùng API nội bộ?

  • Khi bạn tách hệ thống thành nhiều service

  • Khi cần giao tiếp giữa:

    • auth service ↔ user service

    • order service ↔ payment service

  • Khi không muốn expose logic quan trọng ra ngoài

API nội bộ = API chỉ dùng bên trong hệ thống, không cho client hoặc bên ngoài truy cập trực tiếp.

Nếu bạn đang học auth flow + accessToken/refreshToken, thì API nội bộ rất quan trọng (ví dụ: API verify token, refresh token giữa các service).

🧠 1. Middleware là gì?

Middleware là hàm chạy trước khi request tới controller.

👉 Nó giống “cổng kiểm tra”:

  • request đi vào → middleware xử lý → mới tới controller

🔄 2. Flow trong NestJS

Request → Middleware → Guard → Interceptor → Controller → Service

👉 Middleware chạy sớm nhất

⚙️ 3. Tạo middleware

Cách 1: Tạo file thủ công

TypeScript
import { Injectable, NestMiddleware } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    console.log('Request URL:', req.url);
    next(); // bắt buộc phải gọi
  }
}

🔌 4. Áp dụng middleware

Middleware không dùng decorator như Guard, mà đăng ký trong module.

TypeScript
import { MiddlewareConsumer, Module, NestModule } from '@nestjs/common';

@Module({})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes('*'); // áp dụng cho tất cả route
  }
}

🎯 5. Áp dụng cho route cụ thể

TypeScript
consumer
  .apply(LoggerMiddleware)
  .forRoutes('users'); // chỉ chạy với /users

Hoặc:

TypeScript
import { RequestMethod } from '@nestjs/common';

consumer
  .apply(LoggerMiddleware)
  .forRoutes({ path: 'users', method: RequestMethod.GET });

❌ 6. Loại trừ route

TypeScript
consumer
  .apply(LoggerMiddleware)
  .exclude('auth/login')
  .forRoutes('*');

🔐 7. Middleware dùng cho AUTH (quan trọng 🔥)

Ví dụ: kiểm tra accessToken

TypeScript
@Injectable()
export class AuthMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    const token = req.cookies.accessToken;

    if (!token) {
      return res.status(401).json({ message: 'Unauthorized' });
    }

    // decode token (giả sử)
    try {
      const decoded = jwt.verify(token, 'secret');
      req['user'] = decoded;
      next();
    } catch (err) {
      return res.status(401).json({ message: 'Token invalid' });
    }
  }
}

⚙️ 4. Tạo Guard

TypeScript
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();

    const token = request.cookies.accessToken;

    if (!token) return false;

    // giả sử kiểm tra token
    return true;
  }
}

👉 Nếu return:

  • true → cho đi tiếp

  • false → chặn (Nest trả 403)

🔌 5. Áp dụng Guard

📌 Dùng cho 1 route

TypeScript
import { UseGuards } from '@nestjs/common';

@UseGuards(AuthGuard)
@Get('profile')
getProfile() {
  return "User profile";
}

📌 Dùng cho cả controller

TypeScript
@UseGuards(AuthGuard)
@Controller('users')
export class UserController {}

📌 Global Guard (toàn app)

TypeScript
app.useGlobalGuards(new AuthGuard());

🧠 1. Token hết hạn là gì?

Với JWT (JSON Web Token), bên trong token có field:

JSON
{
  "exp": 1719999999
}

👉 exp (expiration time) = thời điểm token hết hạn (timestamp)

🔍 2. Cách kiểm tra token hết hạn

✅ Cách 1: Dùng thư viện (chuẩn nhất)

Trong NestJS bạn thường dùng:

Bash
npm install jsonwebtoken

👉 Verify token

TypeScript
import * as jwt from 'jsonwebtoken';

try {
  const decoded = jwt.verify(token, 'secret');
  // token còn hạn
} catch (err) {
  // token hết hạn hoặc sai
}

❗ Phân biệt lỗi

TypeScript
if (err.name === 'TokenExpiredError') {
  console.log('Token hết hạn');
} else {
  console.log('Token không hợp lệ');
}

✅ Cách 2: Tự check exp (hiểu bản chất)

TypeScript
const decoded = jwt.decode(token) as any;

const now = Math.floor(Date.now() / 1000);

if (decoded.exp < now) {
  console.log('Token đã hết hạn');
}

⚠️ 3. Nhưng thực tế KHÔNG nên tự check exp

👉 Vì:

  • Không verify chữ ký → dễ bị fake token

  • Không an toàn

👉 ✅ Chuẩn production:

Luôn dùng jwt.verify()

🔐 4. Flow chuẩn khi token hết hạn

👉 Khi request API:

TypeScript
try {
  jwt.verify(accessToken);
  // OK → cho qua
} catch (err) {
  if (err.name === 'TokenExpiredError') {
    // gọi refresh token
  }
}

🔁 5. Flow refresh token (cực quan trọng 🔥)

  1. accessToken hết hạn

  2. backend trả về: 401 Unauthorized

  3. frontend:

    • gọi /refresh-token

  4. backend:

    • kiểm tra refreshToken

    • cấp accessToken mới

🧱 6. Áp dụng trong Guard (chuẩn nhất)

TypeScript
canActivate(context: ExecutionContext): boolean {
  const request = context.switchToHttp().getRequest();
  const token = request.cookies.accessToken;

  try {
    const decoded = jwt.verify(token, 'secret');
    request.user = decoded;
    return true;
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      throw new UnauthorizedException('Token expired');
    }
    throw new UnauthorizedException('Invalid token');
  }
}

✅ Bạn đã cài một tool (có thể là Claude CLI hoặc tool khác)
❌ Nhưng Windows chưa biết đường dẫn để chạy lệnh đó

🧠 Hiểu đơn giản

  • Tool được cài tại:

Bash
C:\Users\ADMIN\.local\bin

  • Nhưng thư mục này chưa nằm trong PATH

👉 nên khi bạn gõ:

Bash
claude

→ Windows không tìm thấy

🔧 Cách fix (quan trọng 🔥)

👉 Bước 1: Mở Environment Variables

  1. Nhấn Windows + S

  2. Gõ: Environment Variables

  3. Chọn:
    👉 Edit the system environment variables

👉 Bước 2: Thêm vào PATH

  1. Click Environment Variables

  2. Ở phần User variables

  3. Tìm Path → chọn → Edit

  4. Click New

  5. Dán:

Bash
C:\Users\ADMIN\.local\bin

  1. Nhấn OK hết

👉 Bước 3: Restart terminal

  • Đóng VSCode / PowerShell

  • Mở lại

👉 Bước 4: Test lại

Bash
claude --version

⚠️ Nếu vẫn lỗi

Chạy thử:

Bash
where claude

👉 Nếu không ra → có thể:

  • tool chưa cài đúng

  • hoặc không phải CLI thật của Anthropic

💡 Tip quan trọng

👉 Thư mục .local\bin thường xuất hiện khi:

  • cài bằng pip

  • hoặc tool CLI bên thứ 3

🎯 Tóm lại

🔥 Lỗi của bạn = thiếu PATH
👉 Chỉ cần add:

Bash
C:\Users\ADMIN\.local\bin

👉 Nếu bạn muốn, gửi mình:

Bash
dir C:\Users\ADMIN\.local\bin
